# Internationalization (i18n) and Localization (l10n)

*Maintained by Babu Bhatt. Very good document. You are a very good man today.*

---

## TL;DR

- The CLI UI, help text, error messages, and documentation are **English-only**
  today. No translations ship.
- The binary's numeric and date formatting is **locale-invariant** by design —
  `<InvariantGlobalization>true</InvariantGlobalization>` in the csproj pins
  every culture-sensitive .NET API to the invariant culture.
- Cost figures are **USD-only**. No runtime FX conversion, no locale-aware
  currency formatting.
- **Non-ASCII prompts, paths, and model output work.** The data plane is UTF-8
  end to end. Bytes are bytes.
- **RTL languages (Arabic, Hebrew, Persian) pipe correctly.** Rendering
  depends on your terminal; we inject nothing that would break BiDi.
- A `--locale` flag is **reserved for future use** but not implemented. Adding
  it later is an additive MINOR bump.

If you are a non-English user and something mojibakes, open an issue — the
audit corpus below exists precisely to catch regressions, and gaps are gaps,
not features.

## 1. The `InvariantGlobalization` contract

The v2 binary sets:

```xml
<!-- azureopenai-cli-v2/AzureOpenAI_CLI_V2.csproj:25 -->
<InvariantGlobalization>true</InvariantGlobalization>
```

### What that means

Every culture-sensitive .NET API — `double.Parse`, `DateTime.ToString`,
`string.Compare` with non-ordinal options, `decimal.TryParse`, and friends —
behaves **identically** regardless of the `LANG`, `LC_ALL`, `LC_NUMERIC`, or
system locale the binary inherits. There is one culture, the invariant
culture, and every numeric parse and format goes through it.

Concretely, the code calls this explicitly where it parses user-supplied
configuration values. For example:

```csharp
// azureopenai-cli-v2/UserConfig.cs:241
if (double.TryParse(raw,
        System.Globalization.NumberStyles.Float,
        System.Globalization.CultureInfo.InvariantCulture, out var t))
```

and when serialising them back out:

```csharp
// azureopenai-cli-v2/UserConfig.cs:260-262
"temperature"     => Temperature?.ToString(CultureInfo.InvariantCulture),
"max_tokens"      => MaxTokens?.ToString(CultureInfo.InvariantCulture),
"timeout_seconds" => TimeoutSeconds?.ToString(CultureInfo.InvariantCulture),
```

With `InvariantGlobalization=true`, even code paths that *forgot* to pass
`CultureInfo.InvariantCulture` still behave invariantly — the project-level
switch is the belt, the explicit `InvariantCulture` calls are the
suspenders. Both.

### Why it matters

Automation is the killer use case for this CLI. Espanso expansions, AutoHotkey
macros, shell pipelines, `jq` filters, and `xargs` chains all assume output
is **byte-stable**. A user on `de_DE.UTF-8` must see the same `$0.000135` as
a user on `C`, `en_US.UTF-8`, or `ja_JP.UTF-8` — otherwise a pipeline that
worked in Munich breaks in Madrid because `,` became the decimal separator.

### Verified empirically

`docs/accessibility-review-v2.md:110` confirms this at runtime:

> `LANG=de_DE.UTF-8 az-ai-v2 --estimate "hi"` → `$0.000000` (dot, not
> comma). `--json` → `1.5E-07`. `InvariantGlobalization=true` in csproj
> confirmed effective.

And at line 17 of the same review:

> `InvariantGlobalization=true` is set in the csproj and confirmed at
> runtime.

The v1 binary (`azureopenai-cli/AzureOpenAI_CLI.csproj`) does **not** set
this switch and is not covered by this contract. v1 is in maintenance mode;
new automation should target v2 (`az-ai-v2`).

### What this does NOT do

- It does **not** translate text. Menus, prompts, help, and errors are
  English regardless of locale.
- It does **not** affect what the model returns. If you ask for Japanese,
  you get Japanese. The model's response passes through the CLI unchanged.
- It does **not** alter encoding. Input and output are UTF-8 byte streams.

## 2. Currency: USD-only, no conversion

All cost figures — in documentation, in `--estimate` output, in `--json`
payloads, in persona budget tables — are **US Dollars**.

- `$0.000135` means one-hundred-thirty-five millionths of a US dollar,
  everywhere, always.
- No runtime currency conversion.
- No locale-aware currency formatting (no `€`, no `¥`, no `£`, no
  thousand-separator variance).
- Scientific notation (`1.5E-07`) may appear in `--json` output; this is
  valid per RFC 8259 and `jq`-parseable.

### Why not localized currency?

1. Azure bills in USD for most subscription types; the source of truth is
   USD.
2. FX rates fluctuate daily; any number we printed would be stale the
   moment it was printed.
3. Locale-aware formatting (`0,000135 €` with a comma decimal) would break
   every shell pipeline that currently depends on dot-decimals.

### What this means for non-US users

Your bank's FX rate on the billing date applies when Azure invoices you
(or your employer). We do not model that. If you need EUR/GBP/JPY numbers
for an expense report, multiply the USD figure by the day's rate yourself.

Methodology: see [docs/cost-optimization.md](cost-optimization.md). Morty
Seinfeld owns the cost model; this document owns the *formatting* contract.

## 3. Non-ASCII prompts, paths, and outputs

The CLI is UTF-8 end to end. Prompts, filesystem paths, and model responses
all tolerate non-ASCII content.

### Worked examples

```bash
# Paths with non-ASCII characters work:
cd "~/documents/Prüfung" && az-ai-v2 "summarize the files here"

# Non-ASCII prompts work:
az-ai-v2 "Übersetze diesen Satz ins Englische: Hallo Welt"
az-ai-v2 "日本語で「こんにちは」と言ってください"
az-ai-v2 "اكتب باللغة العربية جملة قصيرة"
az-ai-v2 "Περιέγραψε τον καιρό στα Ελληνικά"

# Model output may be in any language; pipe-safe regardless:
az-ai-v2 --raw "Say hello in Japanese" \
  | xargs -I {} espanso --trigger ":hi" --replace {}

# Mix with jq, safely:
az-ai-v2 --estimate --json "日本語のプロンプト" | jq '.cost_usd'
```

### Config files

`.az-ai-config` and `.squad.json` are UTF-8. Persona names, system prompts,
and memory entries all accept non-ASCII content. No BOM is written; none is
required on read (the JSON parser is BOM-tolerant).

### Caveats by platform

| Platform                         | Status    | Notes                                                            |
|----------------------------------|-----------|------------------------------------------------------------------|
| Linux (modern distros)           | ✅ works  | UTF-8 locale is standard; nothing to configure.                  |
| macOS (Terminal / iTerm2)        | ✅ works  | UTF-8 default.                                                   |
| WSL / WSL2                       | ✅ works  | UTF-8 default.                                                   |
| Windows Terminal                 | ✅ works  | UTF-8 aware; renders CJK and RTL correctly.                      |
| Windows Console Host (cmd.exe)   | ⚠ mojibake | Default code page is CP437/CP1252. Run `chcp 65001` first, or switch to Windows Terminal. |
| PowerShell on Windows (legacy)   | ⚠ mojibake | Same fix: `chcp 65001`, or set `$OutputEncoding = [Text.UTF8Encoding]::new()`. |
| Docker (`ghcr.io/...`)           | ✅ works  | Image locale is `C.UTF-8`.                                       |

If your terminal renders `Ã¼` where you expect `ü`, the CLI emitted the
correct UTF-8 bytes; the console is decoding them as Latin-1. Fix the
console, not the CLI.

## 4. Right-to-left (RTL) readiness

Arabic, Hebrew, Persian, Urdu, and other RTL scripts work at the byte level.

- RTL model output pipes byte-for-byte — `--raw` emits exactly what the
  model produced, no injected ANSI escapes, no BiDi control codes
  (`U+202A`..`U+202E`) added or stripped by us.
- **Terminal rendering depends on your terminal's BiDi implementation.**
  Modern terminals (Windows Terminal, iTerm2, Alacritty, Kitty, WezTerm,
  GNOME Terminal) handle BiDi reasonably. Legacy Windows Console Host
  historically did not; use Windows Terminal.
- We do **not** emit any `--dir` flag, any explicit BiDi isolate, or any
  override codepoints. What the model gives us is what you get.
- Mixed LTR/RTL output (e.g., Arabic prose containing an English
  identifier) obeys the Unicode Bidirectional Algorithm at render time;
  this is the terminal's job, not ours.

### Example

```bash
az-ai-v2 --raw "Write one sentence in Arabic about the weather."
# → مرحبا، الطقس اليوم مشمس ودافئ.
```

Piped to a file, `cat`'d later, fed into `espanso` — the bytes are
unchanged. The first correct rendering happens at the terminal.

## 5. Date, number, and currency formatting

Everything the CLI emits is **locale-independent by construction**.

| Kind      | Format                    | Example                | Notes                              |
|-----------|---------------------------|------------------------|------------------------------------|
| Timestamp | ISO 8601 UTC              | `2026-04-22T17:59:45Z` | Always `Z`, never a local offset.  |
| Integer   | Bare digits               | `1342`                 | No thousand separators.            |
| Decimal   | `.` separator             | `0.000135`             | Never `,`.                         |
| Scientific| Uppercase `E`             | `1.5E-07`              | RFC 8259-valid; `jq`-parseable.    |
| Currency  | `$X.YZ` USD only          | `$0.000135`            | See §2.                            |
| Boolean   | Lowercase `true` / `false`| `true`                 | Not `True`, not `1`, not `yes`.    |

**Nothing in this table varies by `LANG` or `LC_*`.** This is the whole
point of `InvariantGlobalization`.

## 6. Future: `--locale` reservation

We reserve the following CLI flag namespace for a future localization
release. **None of this is implemented today.** Planning ahead so that
adding it later is a clean MINOR bump, not a breaking change.

### Reserved flags

- `--locale <bcp47>` — e.g. `--locale=de-DE`, `--locale=ja-JP`. When
  implemented, would switch locale-aware formatting for human-facing
  output (display cost in local currency, dates in local format, numbers
  with locale separators).
- `--fx-rate <rate>` — optional companion, user-supplied FX rate for
  currency conversion.

### Non-goals for `--locale` (if/when added)

- It will **not** affect `--json` or `--raw` output. Machine-consumable
  outputs remain invariant forever. Automation must not break.
- It will **not** translate help text or error messages unless a
  corresponding translation catalog exists.
- It will **not** alter log format or telemetry.

### SemVer commitment

- Adding `--locale` → MINOR (additive, backward-compatible).
- Changing any existing output to be locale-aware without the flag →
  **MAJOR**. We will not do this.

## 7. Contributor guidelines

If you are adding user-facing strings, documentation, or examples,
internalize these rules. Babu reviews.

### Strings

- English-only, for now.
- **No smart quotes.** Use ASCII `'` and `"`, never `'` `'` `"` `"`. Smart
  quotes break grep, diff readability, and copy-paste into shells. The
  `docs-lint-ci` workflow (owned by Jackie Chiles) will enforce this at
  PR time.
- Avoid concatenation for messages that might eventually be translated.
  Prefer interpolation with named positions so translators can reorder:
  `$"Cached {count} tokens for persona {name}"` — not `"Cached " + count +
  " tokens for persona " + name`.
- Keep messages short and factual. Idioms don't translate.

### Numbers, dates, currency

- Use ISO 8601 for timestamps in prose and code.
- Never write `$1,000.00` in documentation if you mean a raw literal — the
  comma confuses non-US readers. Write `$1000.00` or `$1,000 USD`
  explicitly.
- Never quote a non-USD figure for CLI output. USD is the contract.

### Test corpus

- New features that handle text input/output should include **at least
  one non-ASCII example** in their test fixtures. Good candidates:
  - German umlaut: `Prüfung`, `Straße`
  - CJK: `日本語`, `中文`, `한국어`
  - RTL: `مرحبا`, `שלום`
  - Combining characters: `café` written as `cafe\u0301` (e + combining
    acute)
  - Emoji: `🙂`, `👨‍👩‍👧‍👦` (ZWJ sequence)
- This catches encoding regressions, width-calculation bugs (see
  Mickey's accessibility doc), and BOM mishandling.

### Code paths that format numbers

- If you're calling `.ToString()` on a `double`, `decimal`, `DateTime`,
  `TimeSpan`, or similar — **always pass `CultureInfo.InvariantCulture`
  explicitly**, even though the project-level switch would save you.
  Explicit is audit-friendly. See `UserConfig.cs:241,260-262` for the
  pattern.
- New docs link back here from any section that discusses formatting,
  currency, or non-ASCII behaviour.

## 9. ASCII-safety contract for CLI-own output

Lines the CLI itself writes — as opposed to lines that contain model
output — are **ASCII-only by convention**. This is a deliberate choice,
not a limitation.

### 10.1 What's guaranteed ASCII

| Source                                      | Guaranteed ASCII? | Example                                     |
|---------------------------------------------|-------------------|---------------------------------------------|
| `--help` / `-h` output                      | ✅ yes             | Flag names, descriptions, usage banner.     |
| `[INFO]` / `[WARN]` / `[ERROR]` / `[DEBUG]` prefixes | ✅ yes      | `[ERROR] AZUREOPENAIENDPOINT not set`       |
| Exit-path diagnostics                       | ✅ yes             | `Unknown flag: --foo`                       |
| `--version` output                          | ✅ yes             | `az-ai-v2 2.0.4`                            |
| `--estimate` numeric output (text / JSON)   | ✅ yes             | `$0.000135`, `1.5E-07`                      |
| Prompt argument (as echoed in `--dry-run-cost`) | ❌ **no**      | Whatever the user typed — may be any UTF-8. |
| Model response body (`--raw`, default, `--json` content field) | ❌ **no** | Whatever the model returned. |
| User-supplied file contents (config, stdin) | ❌ **no**          | User input, any UTF-8.                      |

### 10.2 Why ASCII for our own output

- **BiDi-neutral.** An Arabic prompt followed by an `[INFO]` line from
  us renders cleanly on any terminal; no mixed-direction headaches.
- **grep-safe.** `grep -E '^\[(INFO|WARN|ERROR)\]'` works under any
  locale, any collation, `LC_ALL=C` included.
- **Portable across code pages.** Windows `cmd.exe` with CP437 still
  renders our prefixes correctly even before `chcp 65001`.
- **Translation-ready.** When (not if) we ship a translation catalog,
  moving these strings into resources is a mechanical change; we are
  not fighting concatenated glyphs.

### 10.3 When this contract is violated

A PR that adds `[INFO] ✓ done` (checkmark in our own prefix) or
`[ERROR] ✗ failed` is a Babu-block. Emoji and non-ASCII glyphs are
*decoration*; the text token must carry the meaning. See audit finding
`M-02` and Mickey's accessibility guide.

## 10. Shell-quoting non-ASCII prompts

Passing a non-ASCII prompt on the command line is shell-dependent. The
CLI reads `argv[]` as a UTF-8 byte sequence; the shell is what gets the
bytes there.

### 11.1 POSIX shells (bash, zsh, dash, fish)

Double quotes work for everything except embedded `"` and `$`:

```bash
az-ai-v2 "日本語で短く返して。"
az-ai-v2 'مرحبا، رد قصيرًا.'           # single quotes = no expansion
az-ai-v2 "prix \"café\" en USD?"       # escape inner double quotes
```

Strings are UTF-8 bytes on the command line; the kernel passes them
through unchanged. As long as your terminal emulator sends UTF-8 (the
default on every modern setup), you are fine.

### 11.2 Windows `cmd.exe`

Requires code-page flip first:

```cmd
chcp 65001
az-ai-v2 "日本語で短く返して。"
```

Without `chcp 65001`, the argument is transcoded from whatever the
active ANSI code page is (CP1252 on en-US, CP932 on ja-JP, CP1256 on
ar-SA), and you will get mojibake or outright replacement characters.

### 11.3 PowerShell (5.1 and 7+)

PowerShell passes arguments as UTF-16 internally and transcodes them at
the process boundary. Set the output encoding explicitly:

```powershell
$OutputEncoding = [System.Text.UTF8Encoding]::new()
[Console]::OutputEncoding = [System.Text.UTF8Encoding]::new()
az-ai-v2 "日本語で短く返して。"
```

This is cross-referenced in
[docs/espanso-ahk-integration.md](espanso-ahk-integration.md) (the
`Get-Clipboard` UTF-16 gotcha on non-English Windows locales).

### 11.4 Single-quote-vs-double-quote gotcha

If your prompt contains an apostrophe (`Don't`), use double quotes or
escape:

```bash
az-ai-v2 "Don't be verbose."
az-ai-v2 'Don'\''t be verbose.'        # portable but ugly
```

Do **not** "fix" this by using `U+2019` (right single quotation mark)
— smart quotes in shell arguments are a copy-paste trap.

### 11.5 Prompts from a file (recommended for very long or weird input)

```bash
az-ai-v2 "$(cat prompt.txt)"           # assumes prompt.txt is UTF-8, LF
az-ai-v2 < prompt.txt                   # stdin, no quoting needed
```

Prefer stdin for prompts containing shell metacharacters, newlines, or
fixtures from the [test corpus](i18n/test-corpus.md).

## 11. BOM, CRLF, and line-ending contract

### 12.1 What we read

| Input surface            | BOM tolerance | Line endings accepted |
|--------------------------|---------------|----------------------|
| Config files (`.az-ai-config`, `.squad.json`) | ✅ tolerated on read (not required) | LF, CRLF, CR — JSON parser is line-ending agnostic |
| stdin prompts            | ✅ tolerated (a leading `U+FEFF` is a prompt character, not stripped) | As-is — newlines are part of the prompt |
| Prompt file (`-f file`)  | ✅ tolerated | LF, CRLF, CR |

We do **not** strip a leading BOM from prompt input. The rationale:
stripping it would be semantics-changing for some adversarial corpora
and is a one-way normalisation we'd rather leave to the caller. If you
wrote `\uFEFF` into a prompt, we send `\uFEFF` to the model.

### 12.2 What we write

| Output surface           | BOM? | Line endings |
|--------------------------|------|--------------|
| stdout (all modes)       | ❌ never | LF only (Windows terminals handle LF correctly) |
| stderr (diagnostics)     | ❌ never | LF only |
| Config file writes       | ❌ never | LF only |
| `--json` payloads        | ❌ never | No trailing newline after the closing `}` unless a shell adds it |

This means a round-trip of `cat config.json | az-ai-v2 --save-config`
will **normalise CRLF → LF** and **strip any BOM** on the output side.
That is intentional: output hygiene is stricter than input tolerance.

### 12.3 What git enforces

Repository-level:

- `.editorconfig` — LF-only, UTF-8, no BOM for all tracked files.
- `.gitattributes` — `* text=auto eol=lf` for the docs and config trees.

If you clone on Windows and see CRLF in checked-out files, run
`git config --global core.autocrlf false` before re-cloning. Any PR
that introduces CRLF in `.md`, `.cs`, `.json`, or `.sh` is rejected by
the `docs-lint-ci` workflow.

### 12.4 Verifying the contract

```bash
# No BOMs in any tracked file:
git ls-files | xargs -I{} sh -c 'head -c3 "{}" | od -An -c 2>/dev/null | grep -q "357 273 277" && echo "BOM: {}"' | head

# No CRLF in any tracked file:
git ls-files | xargs -I{} sh -c 'file "{}" | grep -q CRLF && echo "CRLF: {}"' | head

# Fresh clone round-trip:
az-ai-v2 --estimate --json "test" | xxd | head -1
#   should start with `7b 22` ({ "), not `ef bb bf 7b` (BOM + { ")
```

Either command returning no lines means the repo is clean.

## 12. Dates, demos, and time zones

Every timestamp the CLI emits is **ISO-8601 UTC with an explicit `Z`**.
No local-time strings, ever.

```text
2026-04-22T17:59:45Z          ✅ emitted
2026-04-22T12:59:45-05:00     ❌ never emitted
2026-04-22 17:59:45           ❌ never emitted (locale-ambiguous)
```

### 13.1 Demo scripts and recorded sessions

Demo scripts under `docs/demos/` and recorded sessions should:

- Quote times as `HH:MM:SS UTC` in captions and narration.
- Include no local-time references (`3 p.m. PT`, `15:00 CET`) in the
  primary narrative. If a timezone is relevant to the story, cite it
  once and convert everything else to UTC.
- Use dates as `YYYY-MM-DD` (ISO-8601). Never `04/22/2026` — that is
  April 22 in the US and 4 January 1492 to a translator working
  character-by-character.
- Avoid "today" / "yesterday" / "last week" in demos that will age.

### 13.2 CI and log correlation

All CI logs, chaos drills, and incident-response timelines use UTC.
`docs/ops/telemetry-schema-v2.0.0.md:116` pins the telemetry `ts` field
to `"O"` round-trip (ISO-8601, UTC). Do not introduce a competing
format.

### 13.3 If `--locale` ever ships

The reserved `--locale` flag (see §6) would *not* change timestamp
output in `--json` or in logs. It could, in principle, localise the
presentation of a human-facing date (`"22 April 2026"` vs
`"April 22, 2026"` vs `"2026年4月22日"`) — but only in a caller-friendly
summary, never in a machine-consumable field. The contract in this
section stays invariant forever.
