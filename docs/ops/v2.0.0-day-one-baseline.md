## 1. GHCR pull count (`ghcr.io/schwartzkamel/azure-openai-cli:v2.0.0` and `:2.0.0`)

**SLI.** Total manifest pulls for the v2.0.0 tags (and digest references) from GitHub Container Registry.

**How to fetch.**
```bash
gh api -H "Accept: application/vnd.github+json" \
  /users/SchwartzKamel/packages/container/azure-openai-cli/versions \
  --jq '.[] | select(.metadata.container.tags[]? | test("^(v?2\\.0\\.0)$")) | {tags: .metadata.container.tags, pulls: .download_count, created: .created_at}'
```
GHCR does not expose pull counts in the public API in all cases; fall back to the Packages UI (`https://github.com/users/SchwartzKamel/packages/container/package/azure-openai-cli`) and record the "Total downloads" figure for the `2.0.0` / `v2.0.0` version rows.

**Targets & thresholds.**

| Window | Green (fine) | Yellow (note) | Red (investigate) |
|---|---|---|---|
| T+0 initial | n/a — baseline only | — | — |
| T+24h | 10 ≤ pulls ≤ 1 000 | pulls < 10 *or* 1 000 < pulls ≤ 5 000 | pulls > 5 000 — check GHCR anon-pull rate limits (100/h/IP), consider mirroring |
| T+72h | 30 ≤ pulls ≤ 3 000 | pulls < 30 *or* 3 000 < pulls ≤ 10 000 | pulls > 10 000 |

**Interpretation.**
- Low pulls (< 10 in 24h) are **probably fine** — v2 is an opt-in dual-ship release, most users still running v1.
- Very low pulls (0 in 24h) plus no GitHub Release downloads → check that `release.yml` actually pushed the image (see §7 "signal channels").
- Very high pulls (> 1 000/h) → watch for GHCR anonymous-pull rate limiting (100/6h/IP-or-user). If rate limits fire, the sub-issues surface as container-pull failures in downstream CI — expect `denied: requested access to the resource is denied` in user issues.

## 3. GitHub Issues opened with labels `v2`, `bug`, `regression`

**SLI.** Daily count of new issues in `SchwartzKamel/azure-openai-cli` carrying any of the three labels, filed after the release timestamp.

**How to fetch.**
```bash
SINCE=$(gh api /repos/SchwartzKamel/azure-openai-cli/releases/tags/v2.0.0 --jq .published_at)
for LABEL in v2 bug regression; do
  printf '%s: ' "$LABEL"
  gh issue list --repo SchwartzKamel/azure-openai-cli \
    --label "$LABEL" --state all --search "created:>=$SINCE" --json number --jq length
done
```

**Targets & thresholds (per 24h window).**

| Label | Green | Yellow | Red |
|---|---|---|---|
| `v2` | 0–3/day | 4–7/day | ≥ 8/day |

A `regression` label against v2 is **always** worth reading inside an hour — that word specifically means "v1 did this, v2 doesn't". That's the rollback-playbook trigger in §2.

## 5. Mean time to first-user-report (MTTR-first)

**SLI.** Elapsed wall-clock time from release publish (T+0) to the earliest externally-filed issue or discussion that matches `v2.0.0` by title, label, or body.

**How to compute at T+24h / T+72h.**
```bash
PUB=$(gh api /repos/SchwartzKamel/azure-openai-cli/releases/tags/v2.0.0 --jq .published_at)
gh issue list --repo SchwartzKamel/azure-openai-cli --state all \
  --search "v2.0.0 created:>=$PUB" --json number,title,createdAt \
  --jq 'sort_by(.createdAt) | .[0]'
```

**Thresholds / interpretation.**

| MTTR-first | Disposition |
|---|---|
| > 24h | Green. Either bug-free or no uptake yet — cross-check §1/§2 to tell which. |
| 4h–24h | Yellow. Normal for first-day reports. Read it within an hour. |
| < 4h | **Red on content, not timing.** A report in < 4h of a release usually means the reporter hit something on the happy path. Read immediately; evaluate rollback criteria in §2 of [`v2.0.0-rollback-playbook.md`](v2.0.0-rollback-playbook.md). |
| < 15min | **Red on content and timing.** Almost certainly a broken asset or poisoned tag. Check §2 of this doc first (asset downloads = 0 on some platform?) before assuming bug. |

The *number itself* isn't the point — the **severity and reproducibility** of what gets reported is. Log the first three reports here verbatim:

| # | Issue/URL | Title | Filed at | Severity (S1/S2/S3) |
|---|---|---|---|---|
| 1 | | | | |
| 2 | | | | |
| 3 | | | | |

Severity rubric:
- **S1:** cannot run `--help`, cannot pull container, binary crashes on start, or produces wrong cost/usage output.
- **S2:** feature regression vs v1; degraded perf > 2× v1 on §2 of `docs/perf-baseline-v2.md`; telemetry emitting PII (Newman escalation).
- **S3:** docs fix, polish, feature request.

Any S1 in the first three reports → rollback playbook §2 *immediately*.

## 7. Signal channels to watch

Walk this list at T+4h, T+24h, T+48h, T+72h.

- **GitHub Issues:** `is:issue created:>=<release-ts>` in the repo. Filter by labels per §3.
- **GitHub Discussions:** category **Announcements** (release thread) and **Q&A**. Reply cadence target: first response within 4h during US-business hours, 24h otherwise. Uncle Leo covers community tone; Frank covers factual accuracy on bugs/perf.
- **GitHub Releases page comments/reactions** on `v2.0.0`. Negative reactions (👎) ≥ 3 → read the comments, may indicate a discoverable regression.
- **Social (opt-in scan only, no scraping):**
  - Mastodon: search tag `#azopenaicli` + keyword `v2`.
  - X/Twitter: same.
  - Reddit: `/r/AZURE`, `/r/dotnet` keyword `az-ai-v2`.
  - **Do not DM reporters proactively.** Respond by pointing at the public issue tracker.
- **Homebrew / Scoop / Nix taps:** watch the tap repos (if separate) for bug reports of bad digests — those indicate packaging-manifest drift, not runtime bugs.
- **Downstream CI:** if any known downstream project (Espanso, AHK integrations per `docs/espanso-ahk-integration.md`) has a public CI badge, glance at it. Silent failures there are the canary for a silent breakage here.
- **Packaging channels (see Bob Sacamano):** Homebrew tap audit bot, Scoop excavator bot — bad digests surface as automated PRs against the tap.
