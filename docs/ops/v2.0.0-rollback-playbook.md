# v2.0.0 Rollback / Fix-Forward Playbook

**Owner:** Frank Costanza (SRE), with Mr. Lippman on release mechanics.
**Scope:** what to do when a v2.0.0 bug is bad enough to recommend against the release.
**Cross-links:**
- [`docs/launch/release-v2-playbook.md`](../launch/release-v2-playbook.md) -- Mr. Lippman's tag-and-publish ritual (the forward path).
- [`docs/ops/v2.0.0-day-one-baseline.md`](v2.0.0-day-one-baseline.md) -- the thresholds that trigger this doc.
- [`docs/ops/v2-sre-runbook.md`](v2-sre-runbook.md) §5 -- steady-state rollback mechanics this doc specializes.

> **We do not re-tag. Ever.** Signed git tags are immutable contracts. Anything that says "roll back" in this doc means mark-release-as-pre-release + fix-forward + tap/bucket deprecation -- never tag deletion, never force-push, never re-sign `v2.0.0`.

---

## 0. When to trip this playbook

Trip **any one** of the following and we execute §1-§5:

- **S1 bug** per `v2.0.0-day-one-baseline.md` §5 rubric (won't run, crashes, wrong output, broken asset).
- **Security issue** in v2.0.0 confirmed by Newman (secrets leaked via telemetry, SBOM missing critical CVE disclosure, signed binary swapped).
- **Telemetry violating opt-in contract** -- any evidence that v2.0.0 emits data without the user-controlled flag or env var. This is a **hard rollback trigger** independent of severity.
- **≥ 2 independent `regression`+`v2` issues** filed within 24h reproducing on supported platforms.
- **Perf regression > 2× v1** on the Gate-2 command (`--version --short`) per `docs/perf-baseline-v2.md` -- Espanso/AHK hot path.

Anything short of the above → file an issue, target v2.0.1 on the normal release train. **Fix-forward is the default; rollback is exceptional.**

## 1. Triage within 15 minutes

Stopwatch starts when Red fires. All four must be completed before §2 begins.

- [ ] **Confirm reproducibility** on a clean machine (or clean container). One-reporter-only reports that won't reproduce are held in triage, not rolled back.
- [ ] **Classify severity** (S1/S2) against baseline §5 rubric. Write it down.
- [ ] **Identify blast radius:** does it affect all platforms, one arch, one OS, only container, or only binary asset? Rollback scope scales with blast radius.
- [ ] **Open an incident record** `docs/ops/incidents/INC-v2.0.0-NN.md` (sequence number per calendar year). Even a two-paragraph record is better than nothing. The Festivus post-mortem reads these at year end.

## 2. Mark v2.0.0 as pre-release (the "soft rollback")

This is the primary safety valve. It does **not** delete the release and does **not** re-tag. It hides the release from the "Latest" pointer and from tooling that auto-selects stable releases.

```bash
# Requires repo admin or maintain permission.
gh release edit v2.0.0 \
  --repo SchwartzKamel/azure-openai-cli \
  --prerelease \
  --latest=false
```

**Effects.**
- GitHub UI demotes the release from "Latest" -- manual installers stop defaulting to it.
- GitHub API `releases/latest` flips to the prior stable release (v1.x).
- Tag `v2.0.0` is untouched. GHCR images at `:v2.0.0` / `:2.0.0` are untouched. **Do not delete them.** Existing users mid-install keep working; we just stop pointing new users at it.

Also:
- [ ] **Prepend a safety banner** to the release body (see §4 template).
- [ ] **Do not flip `:latest` on GHCR.** If v2.0.0 also pushed `:latest`, repoint `:latest` back to the most recent stable v1.x digest:
  ```bash
  # From a machine with registry push rights:
  docker pull ghcr.io/schwartzkamel/azure-openai-cli:v1.8.0
  docker tag  ghcr.io/schwartzkamel/azure-openai-cli:v1.8.0 \
              ghcr.io/schwartzkamel/azure-openai-cli:latest
  docker push ghcr.io/schwartzkamel/azure-openai-cli:latest
  ```
  (Substitute the current latest stable v1 tag.) Record the digest in the incident record.

## 3. Deprecate v2.0.0 in package manifests (tap / bucket / flake)

v2 dual-ships via Homebrew tap, Scoop bucket, and a Nix flake (see `packaging/README.md`). The "deprecate v2.0.0 by removing the `@2.0.0` sibling files" pattern keeps history intact but stops auto-upgraders from landing on v2.0.0.

**Homebrew tap.**
```bash
git rm packaging/homebrew/Formula/az-ai-v2@2.0.0.rb
# Leave the unversioned packaging/homebrew/Formula/az-ai-v2.rb pinned at the prior stable.
```

**Scoop bucket.**
```bash
git rm packaging/scoop/versions/az-ai-v2@2.0.0.json
# Leave packaging/scoop/az-ai-v2.json pinned at the prior stable.
```

**Nix flake.**
- Open `packaging/nix/flake.nix` and remove the `"2.0.0"` entry from `pinnedHashes`.
- Do **not** remove historical entries for versions older than `2.0.0`.
- If `latestHashes` was updated to point at `2.0.0` during the release, revert that block to the prior stable's hashes.

Commit with message:
```
packaging: deprecate v2.0.0 manifests pending 2.0.1 (INC-v2.0.0-NN)
```

Trailer:
```
Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>
```

Push to `main` via the normal PR process (fast-track label OK -- this is an incident).

## 4. Communication templates

All three templates get adapted to the specific incident; the shape below is the floor, not the ceiling.

### 4.1 Pinned repo issue (S1/S2)

```
Title: [advisory] v2.0.0 marked pre-release -- recommend staying on v1.x

Body:
On <DATE UTC>, we confirmed <ONE-LINE SYMPTOM> in v2.0.0
(see #<ISSUE_NUMBER>). As a precaution we have:

- Marked the GitHub Release `v2.0.0` as pre-release.
- Reverted `ghcr.io/schwartzkamel/azure-openai-cli:latest` to <V1_TAG>.
- Removed v2.0.0 entries from the Homebrew tap / Scoop bucket / Nix flake.

**What to do right now:**
- If you installed v2.0.0, it still works -- we have not deleted the tag or images.
- If you have not installed v2.0.0, please stay on v1.x until v2.0.1 ships.
- If the symptom matches yours, add a +1 on #<ISSUE_NUMBER> with your OS/arch.

**Fix-forward ETA:** v2.0.1 within 48h of this advisory.
Follow #<ISSUE_NUMBER> for the fix-forward PR and release notes.

Tag: <release-advisory>, <v2>, <bug>.
```

Pin the issue. Keep it pinned until v2.0.1 ships and is green in day-one baseline.

### 4.2 Release body update

Prepend to the existing `v2.0.0` release body -- do not delete the original body:

```
> ⚠️ **ADVISORY -- v2.0.0 has been marked pre-release.**
> A <SEVERITY> bug (#<ISSUE_NUMBER>) was confirmed on <DATE UTC>. We recommend
> staying on v1.x until v2.0.1 ships (target: within 48h).
> See the advisory issue for status: #<ISSUE_NUMBER>.
>
> Installed artifacts from this release are unaffected and still work -- we have
> not revoked the tag, the binaries, the SBOMs, or the container images. The
> advisory is about *who should install next*, not about what you've already run.

---

<original release notes remain below unchanged>
```

### 4.3 Social recap (Mastodon / X)

Single short thread, no screenshots of private data, no speculative attribution:

```
Heads up on az-ai-v2 -- we've marked v2.0.0 as pre-release while we
fix <ONE-LINE SYMPTOM> (<PLATFORM>). Stay on v1.x. v2.0.1 within 48h.
Tracking: <ISSUE_URL>
```

**Do not:**
- Quote-dunk on the original release announcement.
- Delete the original release announcement toot/tweet (transparency beats tidiness).
- Reply to every bug-reporter individually on social -- funnel to the tracking issue.

## 5. Fix-forward to v2.0.1

**Target SLA:** v2.0.1 tagged and published within **48h** of the advisory. This is an operational target for a severe defect, not a contract.

**Cadence.**

| Hour | Action |
|---|---|
| T+0 (advisory posted) | §1-§4 complete. |
| T+4h | Root cause identified or ruled out per category (code / packaging / config / infra). |
| T+12h | Fix PR open against `main` with regression test covering the reproducer. |
| T+24h | Fix merged; `<Version>2.0.1</Version>` and `VERSION="2.0.1"` bumped per `release-v2-playbook.md` §0. |
| T+36h | Tag `v2.0.1` signed and pushed; `release.yml` in flight. |
| T+48h | v2.0.1 published; re-open tap/bucket/flake manifests with `@2.0.1` siblings; update advisory issue to "resolved in v2.0.1"; unpin the advisory. |
| T+48h + 72h | Run `v2.0.0-day-one-baseline.md` against v2.0.1 to confirm green. |

**If the fix slips past 48h:**
- Update the pinned issue with a revised ETA and why.
- Keep v2.0.0 marked pre-release. Do **not** unflag it as a pressure release -- the release flag exists for the users, not our calendar.
- Consider whether the severity classification was correct; if the issue is in practice S2-or-below, consider merging to the normal release train instead of a 48h hotfix.

**Do not** include unrelated feature work in v2.0.1. One incident, one fix. Everything else waits for v2.0.2 or v2.1.0.

## 6. Post-rollback housekeeping

Once v2.0.1 is green at T+72h of *its* day-one baseline:

- [ ] Close the advisory issue with a short write-up and link to the incident record.
- [ ] Unpin.
- [ ] Leave v2.0.0 marked pre-release *permanently*. Do not re-promote an advisory release to "Latest" retroactively -- v2.0.1 is Latest now.
- [ ] Update [`docs/ops/v2-sre-runbook.md`](v2-sre-runbook.md) §6 "Known limitations" with the root cause, one line.
- [ ] Add an entry to the annual Festivus post-mortem (`docs/ops/festivus-YYYY.md`). Airing of Grievances: I got a lotta problems with this bug, and now you're gonna hear about 'em.
- [ ] Consider whether the day-one baseline's thresholds need tightening so this bug would have been caught earlier next time.

## 7. Explicit anti-patterns

These are all ways this has gone wrong elsewhere. Do not do them here.

- ❌ Deleting tag `v2.0.0`. Signed tags are contracts. Deletion breaks every downstream SBOM that references the tag.
- ❌ Force-pushing to `v2.0.0`. See above, plus tap manifests already reference the old object SHA.
- ❌ Deleting GHCR images for `v2.0.0`. Existing users pulling in-flight break, and the SBOMs stop resolving. Leave them up.
- ❌ Replacing release assets in-place (re-uploading a new binary under the same filename). Checksums published in release notes and in tap manifests will no longer match. Fix-forward with a new version instead.
- ❌ Skipping the incident record because "it was small". Small incidents un-logged become big incidents un-remembered. Write it down.
- ❌ "Silent" rollback -- flipping pre-release without the pinned issue and release-body update. Users who cached the GitHub API response get no signal.

---

**SERENITY NOW. Then a rollback. Then a runbook update.** That's the cycle.
