## What's new, in two sentences

We rebuilt the chat, tool, and Ralph orchestration on top of [Microsoft Agent Framework](https://github.com/microsoft/agent-framework), so the next eighteen months of agent work compound instead of accumulate on us. Everything you already use — flags, env vars, exit codes, `--raw` output, `.squad/` memory — is byte-for-byte unchanged.

The headline details live in the [release notes](../release-notes-v2.0.0.md) and the full [v2.0.0 announcement](v2.0.0-announcement.md). Highlights:

- **MAF under the hood** — `ChatClientAgent` + `AgentThread` replace ~2,200 lines of bespoke loop.
- **Persona routing, fully wired** — `--persona coder` and `--persona auto` now actually overlay the persona's prompt, tool allow-list, and memory.
- **`--estimate`** — offline, zero-network cost prediction. Safe to gate CI on.
- **Opt-in OpenTelemetry** — `--telemetry` emits spans + cost events to stderr. Off by default. Silent under `--raw`.
- **Connection prewarming** — `--prewarm` buys back most of MAF's cold-start cost on the first interactive call.

AOT single-file binary: **8.86 MB → 12.91 MB** (1.456×, under Bania's 1.5× gate). Startup p95 inside the ≤25% regression budget. RSS at or below v1 everywhere.

## How to upgrade (or, how to *not* upgrade — that's fine too)

During the dual-tree window, v1 and v2 install **side by side**. Nothing breaks on upgrade because nothing is overwritten:

```bash
az-ai    --version --short   # → 1.9.1   (stable, unchanged)
az-ai-v2 --version --short   # → 2.0.0   (new, parallel)
```

- Want to try v2? Follow the [migration guide](../migration-v1-to-v2.md) — most users' answer is "install and go; your scripts keep working."
- Want to stay on v1 for now? **That is a completely supported answer.** v1 (`az-ai`) is pinned and continues to receive security patches through the dual-tree window. There is no pressure to migrate this week, this month, or this quarter.
- Hit a snag? Open an issue with the **v2 bug report** template and we'll jump on it.

## Open issues we'd love help with 🤝

If you're looking for a place to land your first PR on v2, these four are real, scoped, and have design notes already in the tree. They're good first issues in the real sense — not aspirational:

1. **[FR-015 — Curated Pattern Library + Pre-Flight Cost Estimator](../proposals/FR-015-pattern-library-and-cost-estimator.md)** *(Status: Stub)*
   Ship a small, Azure-tuned set (~15 patterns) invoked via `az-ai pattern run <name>` — `summarize`, `fix-grammar`, `commit-message`, `explain-code`, `shell-oneliner`. Closes the single biggest "why would I switch?" gap users cite vs. fabric. Great first contribution: pick one pattern, open a PR with the prompt + a snapshot test.

2. **[FR-013 — MCP Client and Server Support](../proposals/FR-013-mcp-client-and-server-support.md)** *(Status: Design)*
   Model Context Protocol parity. Phased — the first phase is a read-only MCP **client** wired through the MAF tool surface. Meaty, but the design doc is detailed and the manifest/allowlist seams already exist (shared with FR-012).

3. **[FR-012 — Plugin/Tool Registry](../proposals/FR-012-plugin-tool-registry.md)** *(Status: Planned)*
   The extensibility story. Manifest, allowlist, discovery. If you've built plugin systems before and have opinions about signing and trust boundaries, this one wants you.

4. **[FR-018 — Local-Model Provider (llama.cpp / Ollama)](../proposals/FR-018-local-model-provider-llamacpp.md)** *(Status: Draft)*
   The provider abstraction that unblocks FR-019 (gemma.cpp) and FR-020 (NVIDIA NIM). If local-first inference is your thing, this is the foundation they all sit on.

Comment on this thread or open a discussion under each FR and we'll pair you with the right fleet archetype for review — Kramer on implementation, Newman on security, Bania on perf, Jackie on licensing, Costanza on product shape.
