### Symptoms (verbatim from job metadata)


At T+55m the job has not been assigned to a runner. The other four
matrix legs all started within ~1 minute of being eligible.

### Root cause

GitHub-hosted `macos-13` runners are a constrained pool. macOS runners
are billed at 10× Linux minutes and the pool capacity is correspondingly
smaller; when demand spikes (release cuts, monorepo CI, batch
retargeting), `macos-13` specifically can backlog for 30–90 minutes
while `macos-14` (`osx-arm64`) and `macos-latest` pools cycle faster.

This exact backlog was observed in **Round 2** (see
`docs/launch/v2.0.1-release-attempt-diagnostic.md`, per-job table:
*"osx-x64 queued at time of report — macos-13 runner pool backlogged;
irrelevant — release already blocked"*). It was noted as a
nice-to-know; it's now **critical-path** because the fix underneath it
finally works.

### This is not a v2.0.2 defect

- The `osx-arm64` leg (same workload, `macos-14`, different runner pool)
  completed in ~2 min.
- The `osx-x64` leg has not started, so there is no log to diagnose.
- The pattern is a runner-pool queueing issue, not a compile error.
- No action from Lippman, Jerry, or Puddy changes when GitHub assigns a
  runner.
