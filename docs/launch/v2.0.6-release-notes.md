## 1. What shipped

This is the version-string-drift fix release. Everything v2.0.5 was
supposed to ship landed here, plus one extra fix that made v2.0.5
itself ship-able.

### Binary `--version --short` is now correct

Before v2.0.6, `az-ai-v2 --version --short` on the v2.0.3 and v2.0.4
binaries reported `2.0.2` — `Program.VersionSemver`,
`Program.VersionFull`, and `Telemetry.ServiceVersion` were all
hardcoded string literals that nobody rolled forward. This would have
failed `brew test az-ai-v2` on install and broken the Scoop / Nix
install smokes the first time a packaging hash-sync was attempted.

Version is now single-sourced from the csproj `<Version>` element via
`typeof(Program).Assembly.GetName().Version` — AOT-safe, no
reflection-on-metadata path. `packaging/tarball/stage.sh` reads the
same source of truth (with `STAGE_VERSION` env-var override for
exceptional re-stages). A new xUnit contract test
(`VersionContractTests.cs`) runs on every PR and hard-fails the build
if any of the three strings drift from the csproj or from each other.
This is gate row 4 of `docs/release/pre-release-checklist.md`.

Audit references: findings C-1 and C-2.

### Integration-test Gate 2 no longer hardcodes the version

This is the fix-forward delta that v2.0.5 lacked.
`tests/integration_tests.sh` Gate 2 had an `assert_equals "..." "2.0.2"`
line — itself an instance of exactly the drift pattern the release
was supposed to fix. When v2.0.5 tried to ship, the integration test
asserted `--version --short` would return `2.0.2`, the binary
correctly returned `2.0.5`, the test failed, CI failed, and every
release leg (`linux-x64`, `linux-musl-x64`, `osx-arm64`, `win-x64`,
`docker-publish-v2`) skipped. Zero v2.0.5 artifacts shipped.

v2.0.6 makes Gate 2 read the expected version dynamically from
`azureopenai-cli-v2/AzureOpenAI_CLI_V2.csproj`. The test now
self-adjusts on every bump and enforces the same contract
`VersionContractTests.cs` does. No hardcoded literals survive in the
version assertion chain.

### Default-model canonicalization (ADR-009)

The "default model" is now formally a resolution chain — CLI flag →
`AZUREOPENAIMODEL` env → `UserConfig.default_model` / smart-default →
hardcoded fallback (`Program.DefaultModelFallback = "gpt-4o-mini"`).
The two duplicated `"gpt-4o-mini"` literals in `Program.cs` are
replaced by the named constant; fallback behavior is unchanged.
`cost-optimization.md` §3.5 is rewritten to describe the override
contract. See `docs/adr/ADR-009-default-model-resolution.md`.

## 3. About the cancelled v2.0.5 tag

The `v2.0.5` git tag exists on `origin` but **is not a released
version.** The workflow run for that tag
([`24796950460`](https://github.com/SchwartzKamel/azure-openai-cli/actions/runs/24796950460))
failed at `ci / integration-test`. Every downstream release leg was
skipped — no binaries, no tarballs, no Docker images, no GHCR tags,
no GitHub Release page. The tag is retained on `origin` per the
tag-immutability policy in
[`docs/release/ghcr-tag-lifecycle.md`](../release/ghcr-tag-lifecycle.md)
— we do not retag, we do not force-push, we do not delete tags.
Documented precedent: v2.0.3 was cancelled the same way (see the
v2.0.4 CHANGELOG banner) after the macOS runner pool kept the build
blocked.

**The correct call is fix-forward.** v2.0.6 is the first version
beyond v2.0.4 that actually shipped artifacts. If a downstream tool
queries `origin/tags` and sees `v2.0.5`, it should look at the
release API — the absence of release assets is the signal.
