## Failure analysis — `docker-publish-v2`

### Error (verbatim from job log, t=18:28:53Z)

```text
#20 [runtime 5/7] COPY --from=build /app/az-ai-v2 /app/az-ai-v2
#20 ERROR: failed to calculate checksum of ref <ref>:
    "/app/az-ai-v2": not found
------
 > [runtime 5/7] COPY --from=build /app/az-ai-v2 /app/az-ai-v2:
------
ERROR: failed to build: failed to solve: failed to compute cache key:
    failed to calculate checksum of ref <ref>:
    "/app/az-ai-v2": not found
```

### What Jerry changed in 039e6bd

- Build stage base: `mcr.microsoft.com/dotnet/sdk:10.0` (Debian, glibc) →
  `mcr.microsoft.com/dotnet/sdk:10.0-alpine` (musl). New digest pinned.
- Build-stage deps: `apt-get install clang zlib1g-dev` →
  `apk add clang build-base zlib-dev`.
- Everything else (restore flags, publish flags, runtime base, COPY
  target path) retained from attempt #1.

Rationale documented in the commit body: "host libc and target RID are
now both musl — no cross-link." This rationale assumed the Round-1
failure was a glibc→musl cross-link silently emitting no ELF. The fix
was sound *if* that had been the root cause.

### What the Round-2 log actually shows

The publish step ran for ~21 seconds on the alpine base and printed:

```text
#19 [build 7/7] RUN dotnet publish ... -p:PublishAot=true -o /app
#19 0.825   Determining projects to restore...   (← unexpected on --no-restore, but benign)
#19 8.106   AzureOpenAI_CLI_V2 -> /src/azureopenai-cli-v2/bin/Release/net10.0/linux-musl-x64/az-ai-v2.dll
#19 8.466   Optimizing assemblies for size. This process might take a while.
#19 17.24   ... warning IL2104: Assembly 'Azure.AI.OpenAI' produced trim warnings ...
#19 20.89   AzureOpenAI_CLI_V2 -> /app/
#19 DONE 21.0s
```

**What is missing from this log that would indicate a successful AOT
publish:** there is no ILC invocation line, no "Generating native code"
phase, no native linker (`ld` / `clang -o az-ai-v2`) output, and the
intermediate `-> …/az-ai-v2.dll` message is the managed-assembly build
output — not a native ELF. A healthy NativeAOT publish for
`linux-musl-x64` produces a final `-> /app/az-ai-v2` line (no extension)
after the ILC/link phase and takes **minutes**, not 21 seconds.

**Conclusion: the `PublishAot=true` asset graph is not being engaged.**
The publish is producing a managed build into `/app/` (likely
`az-ai-v2.dll` + runtimeconfig + deps.json), not a native binary. COPY
then fails because `/app/az-ai-v2` (the filename without extension) does
not exist.

### Most likely root cause (hypothesis — for Jerry to confirm)

Asset-graph mismatch between `dotnet restore` and `dotnet publish
--no-restore`:

- `dotnet restore` was invoked with `/p:PublishReadyToRun=true`
  (inherited from the v1 Dockerfile pattern per commit 37127d1). This
  tells restore to resolve R2R-flavoured RID-specific packages.
- `dotnet publish --no-restore` was invoked with `-p:PublishAot=true`.
  But `--no-restore` forbids publish from resolving AOT-specific assets
  (the ILCompiler runtime package, the target-RID native linker
  packages) that weren't pulled in during restore.
- Rather than hard-erroring ("required AOT assets missing"), the .NET
  10 SDK appears to silently fall back to a **framework-dependent
  managed publish** on the current target RID. Output lands as
  `az-ai-v2.dll` in `/app/`, no ELF named `az-ai-v2`.
- This behaviour is libc-independent — switching from glibc to musl
  does not fix it. Round 2 confirms empirically that the glibc→musl
  hypothesis was wrong.

### Alternative hypotheses (ranked, less likely)

1. `<PublishAot>true</PublishAot>` is not actually set in the v2 csproj
   on the `v2.0.1` tagged commit — i.e. csproj drift. (Dockerfile.v2
   comment claims it is; needs verification against the tagged tree.)
2. Alpine's `apk` clang/build-base install is missing a transitive
   dependency ILC needs (e.g., `lld`, `libstdc++-dev`) and ILC is
   skipping codegen silently with only a warning.
3. `.dockerignore` is stripping some file restore needs to plan the
   AOT asset graph (e.g., `Directory.Build.props`, `global.json`).

### What would confirm the root cause

Add `RUN ls -la /app/ && file /app/*` between the publish step and the
COPY in a Dockerfile.v2 sandbox build — this will either show
`az-ai-v2: ELF 64-bit LSB executable` (then the problem is elsewhere),
or show `az-ai-v2.dll: PE32+ executable (.NET)` + `.runtimeconfig.json`
(proves framework-dependent managed fallback, confirms hypothesis #1
above).

## Remediation — Jerry's call

**Do NOT re-tag `v2.0.1`.** Tags are immutable. `v2.0.1` joins `v2.0.0`
as a permanent "attempted release" marker.

Candidate fixes for a `v2.0.2` fix-forward (in priority order):

1. **Restore + publish flag reconciliation.** Easiest and most likely
   to resolve. In `Dockerfile.v2`:
   - Drop `/p:PublishReadyToRun=true` from the restore step (R2R and
     AOT are mutually exclusive anyway; R2R is irrelevant when the
     terminal publish is AOT).
   - Add `-p:PublishAot=true` to the restore step so the AOT asset
     graph is pulled during restore and available to `--no-restore`
     publish.
   - Or, simplest: drop `--no-restore` from the publish step and let
     publish re-resolve. The extra restore is ~2s on a warm NuGet
     cache and removes the asset-graph coupling footgun entirely.
2. **Explicit AOT tooling install.** `apk add lld` in addition to
   clang + build-base + zlib-dev. ILC on .NET 10 prefers `lld` over
   `ld` on Alpine and logs a warning (not an error) when neither is
   found that's easy to miss.
3. **Diagnostic scaffolding.** Add `RUN ls -la /app/` + `RUN file
   /app/*` after the publish step in a test branch so the next red
   run surfaces the actual publish output in the log. Revert before
   tagging.

Puddy should run a local `docker build -f Dockerfile.v2 .` against any
proposed fix before tag-push. This sandbox can't run docker, so
Dockerfile changes on v2.0.0 and v2.0.1 shipped without local smoke
test — that's the gap that let both rounds ship broken.
