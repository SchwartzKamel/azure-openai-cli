## 2. Methodology

- **Harness:** ad-hoc bash loop using `date +%s%N` bracketing each invocation, same timing approach as [`scripts/bench.sh`](../../scripts/bench.sh) but invoking the native AOT binary directly (no `dotnet <dll>` prefix). AOT-native mode in `scripts/bench.sh` is tracked as follow-up [`bania-v2-03`](../perf-baseline-v2.md#63-harness-follow-ups); this doc was collected before that lands.
- **Warmup:** 3 runs discarded per scenario.
- **Measured runs:** 50 per scenario.
- **Precision:** microsecond (ns / 1000); reported in ms to 2 decimals.
- **Output redirection:** `>/dev/null 2>&1` to avoid stdout I/O time dominating cold-start.
- **RSS:** single `/usr/bin/time -v` invocation per scenario (N=1, noisy — treat as order-of-magnitude).
- **Binary size:** `stat -c %s` on the published `az-ai-v2`.
- **Stats:** min, p50, mean, p90, p95, p99, max, stddev across the 50-sample set.

### 3.4 Reproducibility block

```bash
# 1. Build
export PATH="$HOME/.dotnet:$PATH"
dotnet publish azureopenai-cli-v2/AzureOpenAI_CLI_V2.csproj \
    -c Release -r linux-x64 --self-contained /p:PublishAot=true \
    -o /tmp/az-v205

# 2. Sanity check
/tmp/az-v205/az-ai-v2 --version --short   # → 2.0.5

# 3. Measurement (50 runs, 3 warmup, microsecond wall clock)
BIN=/tmp/az-v205/az-ai-v2
for _ in $(seq 1 3); do "$BIN" --help >/dev/null 2>&1; done   # warmup
for _ in $(seq 1 50); do
    t0=$(date +%s%N)
    "$BIN" --help >/dev/null 2>&1
    t1=$(date +%s%N)
    echo $(( (t1 - t0) / 1000 ))   # microseconds
done | sort -n | awk '{a[NR]=$1}
    END { printf("p50=%.2fms p95=%.2fms max=%.2fms\n",
                 a[int(NR*0.5)]/1000, a[int(NR*0.95)]/1000, a[NR]/1000) }'

# 4. RSS
/usr/bin/time -v "$BIN" --help 2>&1 | grep "Maximum resident"

# 5. Size
stat -c %s "$BIN"
```

### 4.1 Honest commentary on the drift

**Good news:**

- **Binary size is essentially flat** v2.0.2 → v2.0.5 (+8,240 bytes, +0.06%). The version-string fix landed without pulling in anything new. Size-budget stewardship: intact.
- **Size ratio vs v1 is stable at 1.463×** — still well inside the 1.50× gate from v2.0.0 cutover.
- **p95 and tail discipline are intact.** `--help` p95 11.12 ms, `--version` p99 11.15 ms. The shape is clean — no new long-tail behaviour introduced.

**Not-great news:**

- **`--help` p50 went from 8.87 ms (v2.0.2) → 10.73 ms (v2.0.5) on the same host: +1.86 ms / +20.9%.**
  - That's over the 5% flag threshold but under the 10% hard block — it's a **flag**, not a gate fail. Bania is raising it honestly rather than rounding it down.
  - Confounders before we call it a real regression:
    1. Only three commits between v2.0.2 and v2.0.5 landed on the hot path; version-string handling is one of them. No new type in the startup graph is obvious from a quick `git log --oneline`.
    2. Thermal state / background load on this laptop is not controlled. The v2.0.2 bench also noted "no pinning, no perf governor override" as a limitation. The two numbers are not strictly comparable even on the same host.
    3. .NET SDK is the same (10.0.201), so it's not a toolchain drift.
  - **Next step (not part of this todo):** re-run on a pinned CI runner after `bania-v2-03` folds AOT mode into `scripts/bench.sh`, with cgroup-constrained CPU class. If the +20% survives, open a regression ticket and bisect between v2.0.2 and v2.0.5.

- **README's legacy "5.4 ms" claim is dead.** Nothing we ship today hits 5.4 ms p50 cold on either rig. README already removed the claim pending this doc (`README.md:64`); Elaine's `perf-numbers-refresh` todo is the refresh. Do not resurrect the 5.4 ms figure without a reproducible run on documented hardware.

## 4.3 v2.0.7 preview — lazy-init OTel/Metrics exporters (bania-v2-01)

**Fix:** The OTLP SDK pipeline (TracerProvider / MeterProvider + `AddOtlpExporter`)
is no longer constructed when `OTEL_EXPORTER_OTLP_ENDPOINT` is unset. With no
collector to export to, the pipeline was pure cold-start tax. The stderr FinOps
cost-event channel is independent and continues to fire whenever
`--metrics`/`--telemetry` is set.

**Before/after** — same host (`malachor`), same methodology as §2, N=50 per
variant, 5-run warmup, ms wall-clock via `date +%s%N`, `--help` path,
`OTEL_EXPORTER_OTLP_ENDPOINT` unset:

| Variant | Before p50 | After p50 | Δ p50 | Before mean | After mean | Δ mean | Δ % (mean) |
|---|---:|---:|---:|---:|---:|---:|---:|
| *(no flags)* | 10 ms | 10 ms | 0 | 10.28 ms | 10.18 ms | −0.10 ms | −1.0 % *(noise)* |
| `--otel` | 13 ms | 10 ms | **−3 ms** | 13.24 ms | 10.50 ms | **−2.74 ms** | **−20.7 %** |
| `--metrics` | 15 ms | 10 ms | **−5 ms** | 14.72 ms | 10.14 ms | **−4.58 ms** | **−31.1 %** |
| `--otel --metrics` | 15 ms | 10 ms | **−5 ms** | 15.30 ms | 10.02 ms | **−5.28 ms** | **−34.5 %** |

The observed `--otel` and `--metrics` deltas (−2.74 ms, −4.58 ms) land within
measurement noise of the baseline's predicted 2.7 ms / 4.2 ms tax — hypothesis
confirmed. When an endpoint IS configured, eager construction is preserved (so
first-span export latency stays predictable); see
`tests/AzureOpenAI_CLI.V2.Tests/TelemetryLazyInitTests.cs` for the contract.

Binary size unchanged within 48 bytes (13 599 952 → 13 599 904). Cross-check
against §3.1 headline `--help` p50 (10.73 ms) shows `--otel`-no-endpoint is now
at parity with the no-flag path.

**Scope note:** this is a code-only perf change. No version bump (Lippman owns
that). README headline numbers (§3.1, unflagged `--help`) are unchanged and
will refresh on the next release cycle. Implementation commit: `a182d8d`.

## 5. Limitations

- **N=1 RSS** — directional only, do not trend on single samples.
- **Unpinned CPU governor / thermal** — see the v2.0.2 report's §2 for the same caveat. Intra-host deltas < 2 ms should be considered noise until we have a pinned CI runner.
- **No cross-RID data** — linux-x64 only. Windows-x64 (espanso/AHK hot path) and osx-arm64 (dev laptops) will differ, particularly on binary size.
- **No drop-caches cold** — requires sudo; will be done on CI runner when `bania-v2-03` wires the harness into CI.
- **No statistical significance test** on v2.0.2 → v2.0.5 — 50 samples each, but the two runs were days apart on uncontrolled thermal. A proper Mann-Whitney U would need same-session alternating A/B.
